## Goal

Close the M3 wrap-up commit by running a **live** end-to-end Google OAuth
round-trip against a real Google Cloud Project (not a mock), against the
full DB-backed state + first-seen confirm-gesture + prune-scheduler stack
that landed in T3.0–T3.9. Prove every M3 invariant holds under real

## Summary

Booted `docker-compose.smoke.yml` on a wiped `./data/` with real
`GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` / `GOOGLE_REDIRECT_URI` values
obtained from a freshly-provisioned GCP project. Discovered the frontend
`vite build` was failing (`"dismissModal" is not exported by
onboardingUtils.js`) — stubbed the eight missing onboarding helpers in
`src/public/dashboard-app/src/utils/onboardingUtils.js` as
localStorage-backed no-ops purely to unblock the SPA build, and filed
follow-up `F2` for the product-level decision. Rebuilt the SPA inside the
container, ran `docker:smoke:init --force` to mint a master token,
completed signup-mode and login-mode flows through the browser, and then
ran five scripted phases against the live DB + live server.

All five phases passed their M3-relevant assertions:

- **Phase 1 — token auto-refresh.** Aged the Google token (`expires_at`
  forced to the past), issued a proxy call. Call succeeded, and
  `oauth_tokens.expires_at` was updated to a future timestamp. *First
  attempt false-negatived because the in-memory token cache
  (`TOKEN_CACHE_TTL=5min`) served the stale-but-not-expired entry;
  flushed the cache via container restart and it succeeded on the retry.*
- **Phase 2 — prune scheduler.** Aged every `oauth_state_tokens` row into
  the past, invoked `runPruneOnce({ db })` via a scripted call, confirmed
  all aged rows were deleted. Empty-tick silent at INFO as designed.
- **Phase 3 — real proxy (Calendar + Drive).** Attempted agent-token proxy
  calls to `calendar.googleapis.com` and `drive.googleapis.com`. MyApi
  proxied correctly; Google returned `403 PERMISSION_DENIED` because the
  Calendar + Drive APIs were not enabled in the GCP project. This
  validates that MyApi's proxy path surfaces provider errors faithfully
  — the failure was outside MyApi's responsibility and outside M3 scope.
- **Phase 4 — SSRF guards.** Issued proxy calls with target URLs pointing
  at `http://127.0.0.1/...`, `http://169.254.169.254/...`,
  `http://localhost:4500/...`, and the IPv4-encoded variants
  `0177.0.0.1` / `2130706433`. Every single one was rejected at the
  `isPrivateHost` guard before the request could reach the network. M5 is
  still where the real SSRF unification lives, but M3's
  defence-in-depth holds.
- **Phase 5 — returning-user login skips the confirm gesture.** Performed
  a full authorize → callback round-trip for a user whose
  `oauth_tokens.first_confirmed_at` was already stamped. Callback
  immediately 302'd to `/dashboard/` with the session established — NO
  `oauth_status=confirm_login` in the URL, NO
  `oauth_pending_logins` row created. This is the first-seen key from
  ADR-0016 doing exactly what it was designed for: one gesture ever, not
  one per login.

Three non-M3 follow-up issues were surfaced and filed as `F1` / `F2` /
`F3` in `.context/tasks/backlog/`. The operator directed them to be
worked next, starting with `F3` (the Google consent-screen UX — `LogIn.jsx`
hard-codes `forcePrompt=1` on every login-mode authorize URL, so Google
re-asks for consent every single time).

## Key decisions

- **Split the M3 wrap-up into two commits, not one.** The
  `onboardingUtils.js` stubs are a frontend-build hotfix unrelated to
  M3; mixing them into the M3 wrap-up commit would pollute the blame
  history. Commit A ships only the onboarding stubs with a conventional
  `fix(dashboard):` subject; Commit B is the atomic M3 wrap-up proper
  (provider-subject threading + legacy-export retirement + docs +
  session log + task updates + `F1`/`F2`/`F3` briefs).
- **Google API enablement failures (Calendar + Drive) are not M3 bugs.**
  MyApi proxied correctly; Google refused because the APIs were never
  enabled in the GCP project. Documented as environment configuration,
  not a MyApi defect. Gmail API was enabled and the Gmail proxy
  round-trip worked end-to-end.
- **Cache-flushed auto-refresh test is authoritative.** The first Phase 1
  run was a false negative — the 5-minute in-memory token cache served a
  stale token. After a container restart (cache flush) the refresh path
  fired correctly, `oauth_tokens.expires_at` was updated, and the proxy
  call succeeded. The `TOKEN_CACHE_TTL` is a pre-existing concern for a
  different milestone (token-vault hygiene); not an M3 gap.
- **Defer `F1`/`F2`/`F3` to after the M3 wrap-up push.** Operator
  direction: get M3 to `main` first, then take `F3` next.
- **`F3` is queued for the next work session** — operator called it out
  explicitly as the first follow-up to pick up after the wrap-up push.

(No new ADR promotion from this session — ADR-0016 already captured the
first-seen keying decision, and the `F3` fix will get its own ADR when
picked up.)

## Open questions raised

- **When should MyApi force `prompt=consent` on the provider?** The
  answer belongs in `F3`'s eventual ADR. Working theory: signup-mode and
  explicit-reconsent flows only; plain returning-user logins defer to
  the provider's own grant cache.
- **Should `TOKEN_CACHE_TTL` be shorter, or the refresh path eagerly
  invalidate the cache on `expires_at < now`?** Separate milestone,
  probably M10 (database integrity + audit) or a new token-vault pass.
  Not M3.
