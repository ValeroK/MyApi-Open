# ADR-0010 — Single-codebase SOC2 evidence, env-flag toggles

- **Status.** Accepted
- **Date.** 2026-04-21
- **Decision makers.** repo owner + AI pairing
- **Related.** `plan.md` §0.1 (OQ-10), ADR-0004
- **Tags.** compliance, architecture, audit

## Context

SOC2 (and adjacent frameworks like ISO 27001) require continuous evidence:
access control changes logged, secrets rotated, backups verified, audit trail
immutable. If this evidence only exists in a cloud build, the OSS build is
effectively a second untested product for auditors, and contributors can
accidentally regress the cloud-only path.

Follows directly from ADR-0004 (OSS ↔ cloud parity).

## Options considered

| # | Option | Pros | Cons |
|---|--------|------|------|
| A | Separate SOC2 instrumentation in a cloud-only fork | Simpler per-build | Two audit stories; drift between builds; cloud-only code not stressed by OSS users |
| B | **One codebase, all SOC2 instrumentation present, toggled on by env flags in cloud** | Single audit story; OSS users get the security hardening as a side effect; every commit exercises the instrumentation | Slightly larger OSS binary; feature flags must be correct |

## Decision

**Option B.** All SOC2-relevant features live in OSS and are on-by-default or
off-by-default-with-an-env-flag depending on whether they have operational
cost:

| Feature | OSS default | Env flag | Notes |
|---------|-------------|----------|-------|
| Audit log (`audit_log`, `compliance_audit_log`) | ON | — | Always on; DB-trigger-append-only (M10) |
| Audit log Merkle root publishing | OFF | `AUDIT_EXPORT_ENABLED=1` | Cheap once triggered; cloud opts in |
| Pino structured logging with redactor | ON | — | `LOG_LEVEL`, `LOG_INCLUDE_PII` |
| Prometheus `/metrics` | OFF | `METRICS_ENABLED=1` | Cheap; OSS users can opt in |
| OpenTelemetry traces | OFF | `OTEL_EXPORTER_OTLP_ENDPOINT=...` | Costs bandwidth; cloud opts in |
| Key-rotation scheduler | OFF | `KEY_ROTATION_ENABLED=1` | Requires backup + runbook; cloud opts in |
| Billing + Stripe | OFF | `BILLING_ENABLED=1` | Cloud only in practice |
| SSO / SAML | OFF | `SSO_ENABLED=1` | Cloud + enterprise |

## Consequences

- SOC2 evidence is generated by the same code in every deployment, so a
  single set of compliance tests and audit artifacts covers both OSS and
  cloud.
- `package.json` grows slightly with optional deps (Stripe, SSO libs) — keep
  these `optionalDependencies` or gate their `require()` calls behind the
  env flag so OSS installs don't pull them.
- The audit trail is the same shape everywhere, which simplifies incident
  response runbooks.

## Follow-ups

- Catalogued feature flags belong in `docs/architecture/feature-flags.md`
  (new; part of M14).
- Every new cloud-only feature ADR must enumerate its flag and its OSS default.
- A future ADR will formalize the "optional dep gating" pattern once we've
  used it a few times.
